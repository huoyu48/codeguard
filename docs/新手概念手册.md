## CodeGuard 新手概念手册

跟着模块手册的开发顺序走。每开始写一个新模块前，先在这里把不懂的概念搞明白。

---

## 第 1 周：基础骨架

---

### 1.1 pyproject.toml — 项目依赖与工具配置

**依赖（dependency）**：你的代码要用别人的库（比如 FastAPI），这些库就是你的"依赖"。就像做菜需要盐，盐是这道菜的依赖。`dependencies` 列表就是告诉 pip "帮我装这些东西"。

**pyproject.toml**：Python 项目的统一配置文件。以前要用 setup.py + requirements.txt 好几个文件来管项目，现在一个 pyproject.toml 搞定所有事。它告诉 Python：项目叫什么、要装什么包、代码风格怎么检查。

**pip install -e .**：这个命令会读 pyproject.toml 里的 dependencies 列表，把里面列的包都装上。`-e` 是"可编辑模式"，意思是你的代码改了不用重新安装。

**dev 依赖**：`[project.optional-dependencies]` 里的 dev 组是"开发时才需要装的东西"，比如测试框架 pytest。部署到生产环境时不装这些，可以减小体积。

---

### 1.2 .env / .env.example — 环境变量

**环境变量**：操作系统里存的一些"键值对"。比如你的电脑上有个环境变量叫 `PATH`，告诉系统去哪里找可执行文件。程序运行时可以读取这些值。

**为什么要用 .env**：API Key 这种东西不能写在代码里（会被上传到 GitHub 被人看到），所以要放在一个单独的文件里，然后把这个文件排除在 Git 之外。`.env` 就是这个文件。

**.env.example**：是一个"样板"，提交到 GitHub 告诉别人"你需要配这些变量"。别人 clone 你的项目后，复制一份 `.env.example` 改名为 `.env`，填上自己的 Key 就能跑。

**优先级**：系统环境变量 > .env 文件 > 代码里的默认值。意思是如果你同时设置了系统的 `DEEPSEEK_API_KEY` 和 .env 里的 `DEEPSEEK_API_KEY`，系统那个会赢。

---

### 1.3 .gitignore — Git 忽略规则

**为什么需要它**：Git 默认会追踪所有文件的改动。但有些文件不应该被追踪——`.env` 有密钥不能上传，`.venv/` 有几万个文件上传会疯，`__pycache__/` 是 Python 运行时自动生成的缓存。`.gitignore` 就是告诉 Git "这些别管了"。

---

### 1.4 app/config.py — 配置管理

**配置管理**：把程序里需要调整的值（API Key、数据库地址、日志级别）集中在一个地方管理，而不是散落在代码的各个角落。好处是改配置只改一个地方就行。

**pydantic-settings**：一个帮你管理配置的 Python 库。你写一个类定义"我需要哪些配置项，分别是什么类型"，它会自动从 .env 文件里读取对应的值填进去。如果你 .env 里写的类型不对（比如数字写成了文字），启动时就报错，不会等到运行时才崩。

**@lru_cache**：Python 内置的一个小工具。加在函数上面，意思是"这个函数第一次调用时记住它的结果，之后再调用直接返回记住的结果，不再重新算"。用在 `get_settings()` 上，就是让配置对象只创建一次，后面每次用都是同一个。

**单例**：整个程序里只有一个实例的设计模式。`@lru_cache` 让 Settings 变成了单例——不管你调多少次 `get_settings()`，得到的都是同一个对象。

---

### 1.5 app/models/schemas.py — 数据模型

**Pydantic BaseModel**：你可以把它理解成一个"带自动检查的表格模板"。你定义好"这个表格有哪几列，每列是什么类型"，Pydantic 就帮你检查填进来的数据对不对。比如你定义 `pr_number: int`，有人传了个字符串 "abc" 进来，Pydantic 直接报错。

**为什么要用它**：不用 Pydantic 的话，你得手动检查每个字段是不是合法类型、是不是必填。有了它，FastAPI 自动帮你检查，不合法的直接返回 422 错误码，你不用写一行检查代码。

**Field(default_factory=datetime.now)**：这里有个坑。如果写 `default=datetime.now()`，这个时间在你定义类的那一刻就固定了，以后创建的每个对象都会用同一个时间。用 `default_factory` 的意思是"每次创建对象时才去调 datetime.now 拿当前时间"。

**int | None**：Python 3.10 的新写法，表示"要么是整数，要么是空"。有的 GitHub 事件没有 PR 编号，所以 number 可能是 None。

---

### 1.6 app/main.py — FastAPI 应用入口

**FastAPI**：一个用来写网站后端（API）的 Python 框架。你告诉它"这个 URL 对应哪个函数"，它就帮你把 HTTP 的事情都处理好。特点是快、自动生成文档（访问 `/docs` 就能看到）、自动参数检查。

**路由**：就是 URL 和函数的对应关系。`@app.get("/health")` 的意思是"有人访问 /health 这个地址时，执行下面这个函数"。就像餐厅的菜单——每道菜名对应厨房里的一个做法。

**CORS**：浏览器的一个安全规则。你的前端跑在 localhost:5173，后端跑在 localhost:8000，端口不同，浏览器默认会拦截前端的请求（怕有人偷数据）。CORS 中间件就是告诉浏览器"我允许这几个地址来访问我"，请求就不会被拦了。

**健康检查 /health**：一个很简单的接口，就返回 `{"status": "ok"}`。它的作用不是给人用的，是给 Docker、K8s 这些运维工具用的——它们定期访问这个地址，如果长时间没响应，就知道服务挂了，可以自动重启。

---

### 1.7 app/middleware/webhook_auth.py — HMAC 签名校验

**Webhook**：GitHub 的"主动通知"机制。你可以设置一个 URL 给 GitHub，每当有人提交 PR 时，GitHub 就往你的 URL 发一个 HTTP 请求，把 PR 的信息告诉你。不用你反复去问"有新 PR 吗"。就像快递到了快递员打电话通知你，而不是你每隔五分钟跑去驿站问一次。

**中间件**：在请求到达你的路由函数之前，先经过的一道"关卡"。签名校验是中间件，CORS 也是中间件。就像进机场不管你坐哪个航班，都要先过安检。

**HMAC-SHA256**：解决一个问题——"怎么证明这个请求真的是 GitHub 发来的？"你和 GitHub 提前约好一个密码。GitHub 发请求时用密码算一个"签名"带上，你收到后用同样的密码重新算一遍，两个签名对上了，说明确实是 GitHub 发的（别人不知道你的密码，伪造不出来）。

**时序攻击**：普通的 `==` 比较两个字符串时，发现第一个不同的字符就立刻返回 False。这导致"第一个字符对"比"第一个字符错"多花了一点点时间。攻击者可以反复测量这个微小的时间差，一个一个字符地猜出正确的签名。`hmac.compare_digest` 不管对错都比较完整长度，消除这个时间差。

**Depends（依赖注入）**：FastAPI 帮你提前准备好需要的东西。`body: bytes = Depends(verify_webhook_signature)` 的意思是"在执行我的函数之前，先执行 verify_webhook_signature，把它的返回值作为 body 传给我"。如果签名校验失败（抛出异常），你的函数根本不会被执行。你不用在每个路由函数里手动调一遍校验。

---

### 1.8 app/api/webhook.py — Webhook 接收端点

**APIRouter**：FastAPI 用来组织路由的工具。当路由很多时，不可能全写在 main.py 里。用 Router 可以把相关的路由分到不同文件，最后再统一注册到 app 上。就像大公司有不同部门，每个部门管自己的事。

**HTTP 202 vs 200**：200 表示"事情办完了"。202 表示"你的请求我收到了，但还没处理完"。我们返回 202 是因为代码审查是交给后台慢慢做的，不可能在收到请求的这一刻就完成。

**事件过滤**：GitHub 会在很多情况下发 Webhook（PR 被打开、被关闭、被编辑、被合并...），我们只关心"新 PR 被创建"和"PR 有了新代码提交"（synchronize）这两种，其他的一律忽略。

---

### 1.9 app/tasks/celery_app.py + review_task.py — 异步任务队列

**为什么需要异步**：GitHub 发 Webhook 后等你 10 秒钟，如果你没返回就认为你没收到，会重试。但 AI 审查一段代码通常要 5-30 秒。同步处理一定会超时。所以收到请求后立刻把任务扔进后台队列（耗时 < 0.1 秒），先返回"已接收"，后台慢慢做。

**Celery**：Python 的后台任务框架。你用 `@task` 装饰一个函数，它就变成了"后台任务"。调 `process_review.delay(...)` 时不会真的执行函数，而是往队列里放一条消息："请执行 process_review，参数是这些..."。后台有 Worker 进程不断从队列里取消息来执行。

**Redis（作为 Broker）**：Redis 是一个很快的内存数据库。在这里它充当"队列"——Celery 把任务消息存到 Redis 里，Worker 从 Redis 里取。因为它在内存里运行，所以非常快。

**Worker**：一个独立运行的后台进程，不断从 Redis 队列里取任务执行。如果同时来了 100 个任务，多开几个 Worker 就行，不用改代码。这叫"水平扩展"。

**bind=True**：让任务函数能访问 `self`。有了 self 你就能调 `self.retry()` 让任务在失败后重试，用 `self.request.retries` 看当前是第几次重试。

**指数退避**：重试间隔每次都翻倍（10 秒 → 20 秒 → 40 秒）。为什么不等固定时间？因为如果对方服务器出问题了，你频繁重试只会给它更大压力。间隔拉长，给它喘息恢复的时间。

---

### 1.10 app/services/github.py — GitHub API 封装

**REST API**：一种通过 HTTP 请求来操作远程数据的方式。GitHub 提供了 REST API，你可以用 HTTP 请求来获取 PR 的代码差异、发布评论等，不用手动在网页上操作。

**httpx**：一个发 HTTP 请求的 Python 库。跟 requests 很像，但支持异步（`async/await`）。在 FastAPI 这种异步框架里，用 httpx 发请求时不会卡住整个程序等响应。

**封装**：把复杂的操作包在一个类里，提供简单的接口。比如 `github.get_pr_diff(owner, repo, 42)` 一行代码就能拿到 PR 的代码差异，不用关心底层的 HTTP 请求细节。

## 第 2 周：Agent 审查流水线

---

### 2.1 app/agent/tools.py — Agent 工具集

**Agent（智能体）**：一个"会动手的 AI"。普通 AI 是一问一答——你问它问题，它给你回答，完事。Agent 不一样，它能思考"我需要查什么信息"，然后调用工具去查，看到结果后继续思考，直到得出完整结论。就像医生看病——想一下可能是什么病 → 开个检查单 → 看检查结果 → 继续判断。

**@tool 装饰器**：LangChain 提供的定义工具的方式。你写一个函数，加上 `@tool`，它就成了 Agent 能调用的工具。函数名是工具名，函数开头的注释（docstring）是工具说明——AI 读这个说明来决定什么时候用这个工具。所以说明要写清楚"什么时候该用"。

**AST（抽象语法树）**：Python 自带的一个功能，能把源代码解析成树状结构。比如一个 Python 文件里有哪些类、哪些函数，AST 都能提取出来，而且全程不执行代码（很安全）。代码审查工具用它来分析代码结构。

**圈复杂度**：衡量一段代码"有多复杂"的指标。数一数有多少个 if、for、while、except，每个算 +1。数字越大说明分支越多、越难维护。超过 10 就该考虑拆成更小的函数了。

---

### 2.2 app/agent/pipeline.py — LangGraph Agent 流水线

**LangGraph**：LangChain 团队出的框架，用来编排 Agent 的工作流程。它把 Agent 的思考过程画成一张"流程图"（叫 StateGraph），图上标好"这步做什么、下一步去哪"。

**StateGraph（状态图）**：一张流程图。节点（Node）是每一步要做的事（比如"调 AI 思考"、"执行工具"），边（Edge）是节点之间的连线（执行顺序）。条件边是"根据情况决定下一步走哪条路"。

**ReAct 模式**：Agent 的工作循环——**想**（AI 思考下一步做什么）→ **做**（调用工具获取信息）→ **看**（观察工具返回了什么）→ **再想** → 可能再做 → ... → 觉得够了 → 输出最终结论。

**条件边**：根据当前情况决定下一步。比如 `should_continue` 函数检查 AI 最后一条回复里有没有"我要调用工具"的信号：有就去执行工具，没有就说明 AI 觉得够了，流程结束。

**temperature（温度）**：控制 AI 输出"有多随机"的参数。0 几乎每次都给一样的回答（很确定），1 每次回答可能很不一样（很有创造性）。代码审查需要确定性——同一段代码的问题应该每次都能发现——所以设为 0.2，很低。

**ChatOpenAI 传 DeepSeek**：DeepSeek 的 API 接口格式跟 OpenAI 一样（兼容），所以用 LangChain 的 ChatOpenAI 类就能调 DeepSeek。参数名虽然写着 `openai_api_key`，但实际填的是 DeepSeek 的 Key。

---

### 2.3 app/agent/code_reviewer.py — 代码分析 Agent

**System Prompt**：给 AI 的"角色设定"。比如"你是一个资深 Python 代码审查专家"。这会引导 AI 从代码质量的角度来回答问题，而不是像聊天那样随意回答。

## 第 3 周：多 Agent 并行 + RAG 知识库

---

### 3.1 - 3.2 security_scanner.py / doc_reviewer.py — 专业 Agent

**为什么要拆成多个 Agent**：一个 Agent 要同时关注代码质量、安全问题、文档规范，Prompt 会很长，AI 的注意力会分散，效果变差。拆成三个专业 Agent，每个只关注一个方面，Prompt 更短更聚焦，效果更好。

**并行执行**：安全扫描不需要等代码分析做完，它们可以同时进行。用 LangGraph 的 Send API 把同一份代码 diff 同时发给三个 Agent，它们各做各的，最后把结果汇总。总耗时取决于最慢的那个 Agent，而不是三个加起来。

---

### 3.3 app/services/knowledge_base.py — RAG 知识库

**RAG（检索增强生成）**：解决一个问题——AI 不知道你团队的编码规范。做法是提前把规范存起来，审查时先搜出相关的规范，塞给 AI 当参考。这样 AI 的建议就不是泛泛的"加个注释"，而是"根据团队规范 3.2 条，公共函数必须写参数说明"。

**RAG 的完整流程**：
1. **存**：把编码规范文档按段落切开 → 每段转成向量 → 存进向量数据库
2. **查**：审查代码时，把代码片段也转成向量 → 去数据库里找最相似的几段规范
3. **用**：把找到的规范塞进 Prompt，让 AI 参考

**向量**：把一段文字转成一串数字（比如 1536 个数字）。关键特性：意思相近的文字，转出来的数字串也很接近。"代码要有注释" 和 "函数应该写说明" 虽然用词不同，但向量很像。

**向量数据库（ChromaDB）**：专门存向量的数据库。普通数据库用关键词搜索，向量数据库用"距离搜索"——找出向量空间里离你最近的几条。ChromaDB 是轻量级的，数据存本地文件，开发阶段够用。

**文档切分**：一篇编码规范可能几万字，太长了不适合直接存。切成小段落（比如 500 字一段），每段单独存储，搜索时能更精确地找到相关的规范条目。`chunk_overlap` 是相邻两段之间重叠的字数，防止一句话被从中间截断。

**Embedding**：把文字转向量的过程。需要用一个专门的模型来做（比如 OpenAI 的 text-embedding-3-small）。

---

### 3.4 app/services/review_service.py — 审查业务逻辑

**业务逻辑层**：把"做什么"集中在一起的地方。这个模块负责串联整个审查流程：拿到请求 → 搜知识库 → 组装 Prompt → 调 Agent → 收结果 → 发评论 → 存数据库。它是整个系统的"指挥官"。

## 第 4 周：工程化与可靠性

---

### 4.1 app/middleware/rate_limiter.py — 滑动窗口限流器

**限流**：限制请求频率，防止系统被打爆。比如"每分钟最多 60 个请求"，超过了就返回 429（请求太多）。

**滑动窗口**：想象一个 60 秒宽的"窗口"在时间轴上往前滑动。每次新请求进来时，先删掉窗口外的旧记录，再数窗口里有多少个请求。超过 60 就拒绝。

为什么不用"固定窗口"（每分钟重置计数器）？因为固定窗口在边界处有漏洞——59 秒时来了 60 个请求，下一秒又重置了来了 60 个，等于 2 秒内处理了 120 个。滑动窗口不会有这个问题。

**Redis Sorted Set**：用 Redis 的有序集合来实现滑动窗口。每个请求以时间戳为分数存进去，用 `ZREMRANGEBYSCORE` 删过期的，用 `ZCARD` 数有多少个。

**Redis Pipeline**：把多条 Redis 命令打包成一条发出去，减少网络往返次数。就像你把要买的 4 样东西写在一张清单上一次去买，而不是跑 4 趟超市。

---

### 4.2 app/middleware/circuit_breaker.py — 熔断器

**问题**：如果 DeepSeek API 挂了，你的每次请求都会等它超时。大量请求堆积等待 → 你的服务器资源耗尽 → 你的服务也挂了。一个服务挂了导致连锁反应把整个系统拖垮，这叫"雪崩"。

**熔断器**：借鉴电路保险丝的思路。三种状态：

- **正常（Closed）**：请求正常通过
- **熔断（Open）**：连续失败 5 次后，"保险丝烧了"，所有请求直接被拒绝，不再去调用外部 API
- **试探（Half-Open）**：等 30 秒后，放行一个请求试试。成功了就恢复正常，失败了继续熔断

好处是：下游挂了你不会跟着一起挂，而是告诉用户"审查暂时不可用"。

## 第 5 周：数据持久化与监控

---

### 5.1 app/models/database.py — SQLAlchemy ORM

**ORM（对象关系映射）**：用 Python 类来操作数据库，不用写 SQL。你定义一个 `ReviewRecord` 类，属性对应数据库表的列。创建一条记录就是 `session.add(record)`，查询就是 `session.execute(select(...))`，ORM 自动帮你翻译成 SQL。

**SQLAlchemy**：Python 最流行的 ORM 框架。`DeclarativeBase` 是它的声明式写法——你定义类长什么样，SQLAlchemy 帮你建表。

**AsyncSession**：异步的数据库操作。在 FastAPI 这种异步框架里，如果用同步的数据库操作，每次读写数据库时整个程序都得等。用异步版本，等待的时候可以去处理其他请求。

---

### 5.2 monitoring/ — Prometheus + Grafana 监控

**为什么要监控**：你的服务上线后，你怎么知道它运行得好不好？不能一直盯着看。监控系统帮你持续记录关键数据（每秒多少个请求、有多少报错、响应多快），画成图表，出问题了告警通知你。

**Prometheus**：负责"收集数据"。你的程序暴露一个 `/metrics` 地址，Prometheus 每隔几秒来拉一次数据，存起来。

**Grafana**：负责"展示数据"。连接 Prometheus 作为数据源，把数据画成折线图、仪表盘。你可以一眼看到服务的健康状况。

**RED 方法论**：监控重点看三个指标——Rate（每秒多少请求）、Errors（报错率多少）、Duration（响应要多久）。这三个正常，服务就健康。

## 第 6 周：测试与部署

---

### 6.1 tests/test_health.py — 基础测试

**单元测试**：测试一个最小的独立功能。比如测试"配置的默认值对不对"。不涉及网络、数据库等外部东西。

**集成测试**：测试多个模块组合在一起是否正常工作。比如模拟发一个 HTTP 请求，看从签名校验到任务入队这整条链路是否正确。

**pytest**：Python 的测试框架。写测试很简单——函数名以 `test_` 开头，里面用 `assert` 判断结果对不对，就行了。

**Mock**：测试时用假东西替代真实的依赖。比如测试 Agent 时不想真的调 DeepSeek（要花钱、要联网），就用一个假函数返回预设的回复。

**覆盖率**：你的代码有多少行被测试跑过。80% 覆盖率的意思是 100 行代码里有 80 行至少被一个测试执行过。越高越好。

---

### 6.2 .github/workflows/ci.yml — GitHub Actions CI/CD

**CI（持续集成）**：每次有人 push 代码时，自动跑测试。测试不过就不让合并。保证主分支的代码始终是好的。就像交作业前老师自动帮你检查一遍格式。

**GitHub Actions**：GitHub 自带的 CI 工具。你写一个 YAML 文件描述"装依赖 → 跑测试 → 检查代码"，GitHub 在他们的服务器上帮你执行。

---

### 6.3 docker-compose.yml — 容器编排

**Docker**：把你的程序和它需要的所有环境（Python 版本、安装的包）打包在一起。不管在谁的电脑上运行，结果都一样。解决了"在我电脑上能跑"的问题。

**Docker Compose**：CodeGuard 有 5 个服务（FastAPI、Celery Worker、Redis、数据库、Grafana），手动一个个启动很麻烦。Compose 让你在一个文件里定义所有服务，然后 `docker compose up` 一条命令全部启动。

---

### 6.4 .vscode/launch.json — VS Code 调试配置

**调试**：在代码里"打断点"，程序运行到那里会暂停，你可以查看变量的值、一步一步执行，看程序到底在干什么。比 print 好用一万倍。

**--reload**：热重载。你改了代码保存后，服务器自动重启，不用手动停掉再启动。开发时很方便。

**justMyCode: false**：允许调试进入第三方库的源码。比如你可以打断点看 FastAPI 内部是怎么处理你的请求的，学习框架原理时有用。
